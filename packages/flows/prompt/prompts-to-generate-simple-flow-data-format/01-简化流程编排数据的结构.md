# 简化流程编排数据的结构

## 背景

我们已经实现了一个流程编排设计器，类似Dify、Coze，可以用来可视化的编辑一个流程图，用户的操作主要就是：拖拽创建节点、在节点之间连线、编辑节点的属性。

现在，用户希望可以通过自然语言描述的方式，利用AI自动生成流程图的数据。但是，我们目前的流程数据比较复杂，且其中包含了很多冗余数据，还包含嵌套，是不利于AI理解和生成的。

所以我们希望先简化现有的流程数据结构，让AI生成简化版的流程数据，然后我们再通过一段程序将之转化为设计器实际需要的结构。

参考A2UI等协议，利于AI生成的数据结构应该尽量没有冗余数据，尽量扁平化，尽量不包含嵌套。

在`raw`文件夹下，包含了流程原数据结构的ts类型。在`raw\flow-metadata.ts`中的`FlowMetadata`就是现有的流程编排数据，其它字段都比较简单，只考虑`nodes`字段，它代表流程中的所有节点，本次简化的核心就是简化流程节点的结构。我们目前有7种不同的流程节点，具体的类型定义在`raw\all-nodes.ts`中。

### 关于“函数调用节点”

创建`函数调用节点`数据时，核心是填写`express: MethodInvokeExpr`字段，需要根据用户的实际需要，从包含方法的类型中选择一个方法来填写这个字段。

所有包含方法的类型详情都在文件`raw\all-method-type.json`中，它们都符合`raw\type.ts`中的`Type`类型。

填写`MethodInvokeExpr`时只需要填写以下4个字段：

1. kind：固定为"methodInvoke"
2. typeUrl：要调用的函数所属的类型，对应`typeId`字段
3. methodCode：要调用的函数，对应`methods.code`
4. parameters：函数的入参列表

### 关于“设备调用节点”

创建`设备调用节点`的数据时，你需要知道有哪些设备类型、每种设备类型有哪些可调用的操作、每种操作需要传递的入参列表是什么，这些都需要参考设备元数据列表。设备元数据用来描述某种类型的设备包含哪些事件、哪些可调用的方法。

所有可选的设备元数据都在`raw\all-device-metadata.json`文件中，其中的每一个设备元数据都符合`raw\device.ts`中的`DeviceModel`类型。

在`DeviceCallNode`中，`deviceModelId`字段对应`DeviceModel`中的`modelId`，`deviceId`字段暂不考虑，`deviceAction`对应`DeviceModel`中`actions`的一个`key`，表示要调用的操作。

### 关于“设备事件监听节点”

与`设备调用节点`类似，创建`设备事件监听节点`时，也依赖设备元数据列表`raw\all-device-metadata.json`。

## 初步目标

关注7种节点，简化它们的数据结构。

在`examples\nodes.md`中生成每一种节点的简化后的数据示例，目前该文件中已经包含了几个节点的简化版数据示例，目前的改动思路包括：

1. kind 更名为 type
2. 去掉节点的 code，只生成 id 即可
3. 简化复杂的表达式，只保留最必要的信息
4. 扁平化，循环节点的子节点使用"parentNodeId"字段指向父级

如果现有的简化示例还能优化，你可以直接修改。

## 最终目标

生成一个完整的MD文档，AI阅读了这个文档就知道怎么生成简化版的流程编排数据了。其中要包含一个Json Schema，用于详细规范简化版流程编排数据的结构。
